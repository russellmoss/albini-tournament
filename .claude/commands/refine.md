# /refine — Triage and Apply Council Feedback

> Generated by rstack for **albini-tournament** ()

Apply council review feedback to the implementation plan. Triage every item, auto-apply what's safe, ask about what's ambiguous.

---

## Prerequisites

Both of these must exist:
- An implementation plan (`agentic_implementation_guide.md` or matching pattern)
- Council feedback (`council-feedback.md`)

If either is missing, tell the user what's needed and STOP.

---

## Step 1: Read Everything

Read these files silently:
- The implementation plan
- `council-feedback.md`
- `code-inspector-findings.md` (for construction site verification)
- `data-verifier-findings.md` (for field name verification)
- `pattern-finder-findings.md` (for pattern verification)
- Any prior conversation context about this feature



---

## Step 2: Triage Feedback

For each item in council-feedback.md, classify into one of three buckets:

### Bucket 1: Apply Immediately
Items where the correct fix is determinable without human judgment:
- Construction site corrections (wrong file path, missing file)
- Field name fixes (wrong casing, wrong field)
- NULL handling improvements
- Missing error handling
- Pattern inconsistencies (using wrong pattern vs established ones)
- Phase ordering issues (dependency used before it's created)
- Validation gate updates (wrong or missing validation commands)
- Import/dependency fixes

### Bucket 2: Apply Based on User Answers
Items that require human judgment:
- Business logic interpretation
- Design decisions (UI layout, user flow)
- Calculation formula changes
- Field label/display changes
- Sort order decisions
- Scope questions (include this feature or defer?)
- Architectural tradeoffs

### Bucket 3: Note But Don't Apply
Items to acknowledge but not act on:
- Scope expansion beyond the original request
- Alternative architectural approaches
- Performance optimizations not requested
- Items the user explicitly declined
- Suggestions that contradict project conventions

---

## Step 3: Apply Changes

For each Bucket 1 item:
1. Make the edit to the implementation plan
2. Verify the fix is consistent with the rest of the plan
3. Update validation gates if affected

**Do not apply Bucket 2 items yet.** Collect them for the human input gate.

---

## Step 4: Append Refinement Log

At the end of the implementation plan, append:

```markdown
## Refinement Log — Council Review

### Applied (Bucket 1)
- [item]: [what was changed and why]

### Pending User Input (Bucket 2)
- [item]: [question for the user]

### Noted (Bucket 3)
- [item]: [why it was deferred]
```

---

## Step 5: Self-Review

Before presenting to the user, verify:
- [ ] Phase ordering is consistent (no forward references)
- [ ] All file references still exist in the plan
- [ ] Type changes are reflected in all consuming files
- [ ] Validation commands are correct: `npm run build`, `npx tsc --noEmit`
- [ ] No template variables leaked (no `{{...}}` in output)

---

## Step 6: Report

Tell the user:
- How many items were auto-applied (Bucket 1)
- Present Bucket 2 items as questions (if any) — STOP and wait for answers
- How many items were noted but deferred (Bucket 3)

If there are Bucket 2 items, **STOP** after presenting them. Wait for the user to answer before applying.

If no Bucket 2 items, tell the user: "All feedback applied. The implementation guide is ready for execution."
