## Problem Statement

Run a one-night, in-person caroms billiards tournament for ~15 friends in a "World Cup" format: group stage → 8-player single-elimination knockout. The host runs everything from a phone or laptop on the night. Spectators watch the bracket on their own phones. The visual identity is Steve Albini — groups named after records he produced, blurred album-cover slideshow behind a framed content card.

Goal for night-of: zero arguing about scores, zero confusion about who plays who next, looks great on a phone in a dim bar.

## What Makes This Cool

- The Steve Albini conceit. Group "In Utero" plays Group "Surfer Rosa" feels like the right tribute.
- The album-cover slideshow gives the app a distinct atmosphere — most tournament apps look like spreadsheets.
- The "First Inaugural" framing implies a tradition. If this works, it runs again next year.
- Caroms tournaments rarely get a real bracket UI. This is a small group of friends getting a presentation that feels like a real event.

## Constraints

- ~15 players, exact count known by lock time, no late arrivals after lock.
- Stack fixed: Next.js 15 (App Router), Neon Postgres, Vercel deploy.
- ENV already has Anthropic + Gemini keys (not used in MVP, but available for future features like AI-generated match commentary).
- Album cover images live at `C:\Users\russe\Documents\albini-tournament\images` (currently 3 files; user will collect 20–30 before launch and pre-blur them at build time).
- Mobile-first: host runs admin from a phone, spectators view bracket on phones.
- One-evening event: no need for offline mode, persistent sessions across days, or scaling beyond ~30 concurrent viewers.

## Premises

Confirmed during /office-hours session:

1. **N=15, 3 groups of 5.** 6 group winners (top 2 each) + 2 wildcards = 8 in knockout. "Wildcard" = best-of-rest selected by the same tiebreaker waterfall used inside groups.
2. **Hard registration lock.** State machine: register → lock → draw groups → group play → knockout → done. No late arrivals after lock. If someone flakes after the draw, their group plays with a forfeit — too bad.
3. **Tiebreaker waterfall** (within group AND for wildcard selection):
   1. Games won
   2. Total points scored across group games
   3. Point differential between the tied players (head-to-head)
   4. If 3+ players are tied: re-apply 1–3 to the mini-table among only the tied players
   5. If still tied: random (documented coin flip in the UI)
4. **Per-game normalization is required for wildcard math** when groups are different sizes. With N=15, 3×5, all groups are equal so this doesn't bite — but the algorithm should compute win rate / points-per-game / diff-per-game so it stays correct if N drifts in future tournaments.
5. **Strict seed bracket.** 1v8, 4v5, 3v6, 2v7. Top half: (1v8) vs (4v5). Bottom half: (3v6) vs (2v7). Final winners of top vs bottom. No World Cup–style group-cross seeding. Group-stage rematches in QF are fine.
6. **Multi-tournament schema from day one.** The data model assumes a `tournaments` table; the UI hardcodes "the active tournament" for night-of simplicity.
7. **Auth: admin PIN cookie.** One shared 4-digit PIN unlocks write actions. Public read-only bracket. No NextAuth, no magic links.
8. **Album slideshow: pre-blurred WebPs at build time** (user is collecting 20–30 covers before launch). Dark overlay (`rgba(0,0,0,0.5)`) under the framed content card for text readability.
9. **Polling, not SSE.** Bracket page polls server every 5s. Acceptable lag for one-night event.

## Approaches Considered

### Approach A: Server Actions + Polling (Recommended)
Normalized 6-table schema. Next.js 15 RSC pages, server actions for every mutation. 5s polling on the spectator bracket page. Drizzle ORM. PIN cookie middleware for admin routes. ~12 hours to build.

### Approach B: Server Actions + SSE
Same as A but the bracket page subscribes to a Server-Sent Events route handler. Live push on every match result insert. Requires Upstash Redis pub/sub channel for serverless-safe broadcast. ~16 hours. Better in-room feel; more moving parts and more risk on Vercel free-tier function-duration limits.

### Approach C: Tournament-as-Document
2-table schema with all tournament state in a single `state jsonb` column. Server actions are pure functions over the blob. ~6 hours. Fastest to build, but cross-tournament queries become awkward and the eventual port to normalized rows is a real migration.

### Schema (Drizzle)

```typescript
// 6 tables. Multi-tournament from day one.

tournaments {
  id          uuid pk
  name        text not null               // "First Inaugural Albini Invitational"
  status      text not null               // 'draft' | 'locked' | 'group_play' | 'knockout' | 'done'
  config      jsonb not null              // { group_count: 3, group_size: 5, advance_top: 2, wildcard_count: 2 }
  created_at  timestamptz default now()
  updated_at  timestamptz default now()
}

players {
  id              uuid pk
  tournament_id   uuid fk → tournaments.id (on delete cascade)
  name            text not null
  registered_at   timestamptz default now()
  unique(tournament_id, name)             // no two "Steves" in one tournament
}

groups {
  id              uuid pk
  tournament_id   uuid fk → tournaments.id
  name            text not null           // "In Utero", "Surfer Rosa", "At Action Park"
  position        smallint not null       // 1, 2, 3 — for stable display order
  unique(tournament_id, position)
}

group_assignments {
  id          uuid pk
  group_id    uuid fk → groups.id (on delete cascade)
  player_id   uuid fk → players.id (on delete cascade)
  unique(group_id, player_id)
  unique(player_id)                       // a player is in exactly one group per tournament
}

matches {
  id              uuid pk
  tournament_id   uuid fk → tournaments.id
  phase           text not null           // 'group' | 'qf' | 'sf' | 'final'
  group_id        uuid fk → groups.id     // nullable; only for phase='group'
  bracket_slot    smallint                // nullable; 1..7 for knockout (1-4 = QF, 5-6 = SF, 7 = Final)
  player_a_id     uuid fk → players.id
  player_b_id     uuid fk → players.id
  status          text not null           // 'pending' | 'done'
  played_at       timestamptz
  check (
    (phase = 'group' AND group_id IS NOT NULL AND bracket_slot IS NULL)
    OR (phase != 'group' AND group_id IS NULL AND bracket_slot IS NOT NULL)
  )
}

match_results {
  id          uuid pk
  match_id    uuid fk → matches.id (on delete cascade)
  player_id   uuid fk → players.id
  points      smallint not null           // raw points scored in the match
  won         boolean not null            // true for the winner; ties not allowed in caroms (spec assumes this)
  unique(match_id, player_id)             // one row per player per match (always exactly 2)
}
```

**Indexes**: `(tournament_id, phase)` on matches, `(tournament_id)` on players. The dataset is tiny (~50 rows total per tournament) so additional indexes are unnecessary.

### State Machine

```
draft ─[admin: Lock Registration]─→ locked
locked ─[admin: Draw Groups]──────→ group_play
group_play ─[all group matches done]
           AND [admin: Start Knockout]─→ knockout
knockout ─[final played]──────────→ done
```

The `status` column gates which UI surfaces are reachable. Spectator pages render different content per state ("Registration open — N players signed up" vs. "Group play in progress" vs. "Knockout: round of 8" vs. "Champion: <name>").

### Algorithms

**1. Group draw** (`lib/tournament/draw.ts`)
- Input: array of player IDs, group_count from `tournaments.config`
- Shuffle with crypto-grade RNG (`crypto.getRandomValues` over a Fisher-Yates pass)
- Distribute players round-robin into groups: player[i] → group[i % group_count]
- Generate group names: pick `group_count` unique names from a curated Albini-records list (`['In Utero', 'Surfer Rosa', 'At Action Park', 'Rid of Me', 'Pod', 'Atomizer', 'The Hammer Party', 'Songs About Fucking', ...]`) — shuffle and take first N
- Generate group matches: round-robin within each group. For 5-player groups, that's `C(5,2) = 10` matches per group, 30 total
- All matches start with `status='pending'`; admin clicks each as it's played

**2. Standings + tiebreakers** (`lib/tournament/standings.ts`)
- Compute per-player within a group: `games_played`, `wins`, `points_scored`, `point_diff`
- Sort by waterfall: `wins DESC`, `points_scored DESC`, then resolve ties:
  - For 2-way ties: `head_to_head_diff DESC` (the diff in their direct match)
  - For 3+-way ties: build a sub-table containing only matches between tied players, re-apply `wins → points_scored → diff` over the sub-table
  - If still tied: deterministic random (seeded by tournament_id + player_ids sorted) — same seed always produces the same order, so the UI can show "tied; coin flip resolved in favor of X" without arguing

**3. Wildcard selection** (`lib/tournament/wildcards.ts`)
- After all group matches done, collect all non-top-2 finishers across groups (for N=15: 3 groups × 3 non-top-2 = 9 candidates)
- Compute per-game metrics: `wins / games_played`, `points_scored / games_played`, `point_diff / games_played`
- Sort candidates by per-game waterfall: `win_rate DESC`, `pts_per_game DESC`, `diff_per_game DESC`
- Pick top `wildcard_count` (2 for N=15)
- Deterministic random tiebreak as above if still tied at the cutoff

**4. Bracket seeding** (`lib/tournament/bracket.ts`)
- Combine 6 group winners (top 2 from each group) + 2 wildcards = 8 players
- Assign global seeds 1–8 by overall standing across all 8: `wins DESC`, `pts_per_game DESC`, `diff_per_game DESC`, then deterministic random
- Place into bracket slots:
  - QF1: seed 1 vs seed 8 (slot 1)
  - QF2: seed 4 vs seed 5 (slot 2)
  - QF3: seed 3 vs seed 6 (slot 3)
  - QF4: seed 2 vs seed 7 (slot 4)
  - SF1 (slot 5): winner QF1 vs winner QF2
  - SF2 (slot 6): winner QF3 vs winner QF4
  - Final (slot 7): winner SF1 vs winner SF2
- Insert all 7 matches at once with `status='pending'` and `player_a_id`/`player_b_id` populated for QF; SF and Final get populated as upstream matches complete (server action that posts a match result also updates downstream pending matches)

### UI Surfaces

**Public**
- `/` — landing card. Shows tournament name, status, player count, and a "Watch Bracket" CTA once group play starts.
- `/groups` — live group standings. Three tables (one per group), highlighting top-2 in green and 3rd-place candidates in amber while group play is active. Polls every 5s.
- `/bracket` — knockout bracket. SVG/CSS rendering of the 8-player bracket. Polls every 5s. Champion celebration screen on `done`.

**Admin** (gated by PIN cookie middleware)
- `/admin/login` — PIN entry. POSTs to a server action that sets a signed cookie.
- `/admin` — dashboard. Big buttons that change with state: "Add Player" / "Lock Registration" / "Draw Groups" / "Post Group Result" / "Start Knockout" / "Post Knockout Result".
- `/admin/players` — add/remove players (only when status='draft'); list of registered players.
- `/admin/groups` — list of pending group matches; tap one → score entry (two number inputs + a winner radio); submit → server action.
- `/admin/knockout` — same UI shape as group match entry, but for QF/SF/Final.

### Auth (PIN cookie)

- Admin sets `ADMIN_PIN` env var on Vercel (e.g., `4729`).
- `/admin/login` form posts PIN to a server action; if matches, sets a signed HTTP-only cookie (`admin_pin_ok=1`, signed with `AUTH_SECRET` env var).
- `middleware.ts` checks the cookie on every `/admin/*` route; redirects to `/admin/login` if missing/invalid.
- No user accounts. Cookie expires in 12 hours (covers the night).
- Public pages have no auth.

### Background Slideshow

- 20–30 album covers pre-processed at build time (or pre-deploy):
  - Convert to WebP, max 1920×1920
  - Apply blur in image processing (`sharp({...}).blur(40)`) — bake the blur in, don't rely on CSS `filter: blur()` (CPU-expensive, janky on phones)
  - Output to `public/covers/*-blur.webp`
- Client component `Slideshow.tsx`:
  - Renders a fixed-position container behind everything
  - Cross-fades between two `<img>` elements every 8 seconds
  - Shuffles cover order on mount
  - Preloads next image
- `FramedCard.tsx` wraps page content with:
  - `rgba(0,0,0,0.5)` darkening overlay layer
  - Centered card with subtle border, pure black background, off-white text
  - All page content lives inside `<FramedCard>` so the overlay is consistent

### Polling

- All public pages use a client component `<PollingRefresh interval={5000} />` that triggers `router.refresh()` on a timer
- `router.refresh()` re-runs RSC; minimal payload because RSC streams diffs
- No additional libraries. Stop polling when document is hidden (`document.hidden` check in the poll callback) to save battery

## Open Questions

1. **Album cover licensing for public Vercel URL.** Fair use for a private one-night event is fine in practice, but the URL is technically public. Options: (a) ignore — nobody comes for tournament apps; (b) deploy under an unguessable subdomain (`steve-honors-jed-9k2x.vercel.app`); (c) put the album-cover content behind the admin PIN. Recommend (a) with optional (b).
2. **What happens if a player flakes between draw and group play start?** Spec says no late arrivals — but no spec for early withdrawals. Suggest: admin can mark a player as "withdrawn" before any of their group matches are posted; their pending matches are forfeited (they get 0 wins, opponents get a `won=true` row with `points=0` placeholder). Confirm this matches intent.
3. **Caroms scoring rules — points per match.** Spec assumes integer points scored per player per match and no ties. Worth confirming: is there a fixed game length (first to N points), or do you play to a target and the loser's score is whatever they had when the winner finished? This affects what the score-entry UI should look like.
4. **AI commentary / match write-ups.** ENV has Anthropic + Gemini keys. Out of MVP scope, but if you want a "Champion's Story" page generated from the match log at the end of the night, the data model supports it (the `match_results` table is a complete event log).
5. **Domain name.** `albini-tournament.vercel.app`? Custom domain? Affects how easily the URL is shared on the night.

## Success Criteria

- Host can register all 15 players and lock in under 5 minutes from a phone in a dim bar.
- Group draw is one tap; group match results are 3 taps each (open match → enter scores → submit).
- Spectators can watch live bracket updates on their phones with no install, no auth.
- Zero server errors during the event.
- The slideshow renders smoothly on a 3-year-old iPhone (no jank, no battery panic).
- The bracket page loads in under 1 second on 4G.
- Tomorrow morning, the data is still in Postgres for next year's "Second Annual."
